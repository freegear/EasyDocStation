# 1 화면 구성

Slack과 같은 서비스를 구축할 것이다.
화면은 크게 나누면 3개로 나눌 수 있다.

1) 왼쪽은 Team 및 채널 목록을 보여주는 화면
2) 가운데는 선택된 채널의 대화 내용을 보여주는 화면
3) 오른쪽은 GROQ 서비스를 연동하는 것이다.
4) 제일 위에는 타이틀 바와 함께 로그인 한 사람의 정보가 표시된다.

# 1.1  왼쪽은 Team 및 채널 목록을 보여주는 화면
# 1.1.1 왼쪽은 Team 및 채널 목록을 보여주는 화면

# 1.1.3 Team 및 채널 목록 보여주기

사이트 관리자는 모든 Team 목록과 해당 팀의 모든 Channel 목록을 볼 수 있다.
Team 관리자는 해당 Team의 모든 채널 목록을 볼 수 있다.
사용자는 
- 자신이 등록된 Team의 목록을 볼 수 있다. 
- 선택된 Team 목록에서, 자신이 등록된 채널의 목록을 볼 수 있다. 

새로운 사용자가 등록이 되면, 
- 아무런 Team이나 Channel에 등록되지 않는다.
- 사이트 관리자 , 팀관리자, 채널 관리자가 등록 시켜야지만 등록이 된다.

로그인이 되면
- 화면을 자동으로 한번 Refresh를 하여준다.
- 그래서 보기가 가능한 Team과 Channel을 보여준다.

사용자가 Team에 등록이 되면 해당 팀에 등록된 채널 목록중, 사용자가 등록된 채널만을 볼 수 있다. 
- 	public 채널 전체 + 등록된 채널 을 볼 수 있다.

사이트 관리자는 
- Team 추가 버튼을 볼 수 있다.
- Channel 추가 버튼을 볼 수 있다.

사이트 관리자는 모든 댓글과 게시글을 수정하거나 삭제할 수 있다.
사이트 관리자는 모든 채널을 삭제하거나 수정할 수 있다.
사이트 관리자는 모든 팀을 삭제하거나 수정할 수 있다.

# 11. Local-S3 Mocking 서비스 아키텍처 설계 및 구현

이 서비스는 AWS S3의 역할을 자체 API 서버(Local File Service)와 로컬 디렉토리가 분담하여 수행합니다.
이와 같은 서비스를 지원하는 서버를 만들어주고, EasyDoc이 실행할 때에 자동으로 실행되도록 한다.
Local File을 저장하는 폴더는 

config.json 파일에서 "Object File 폴더" 정보를 기반으로 한다.


1. 주요 구성 요소
클라이언트 (Web/Mobile): 파일을 선택하고 업로드/다운로드하는 사용자 인터페이스. S3 SDK 대신 표준 HTTP 클라이언트(Axios, Fetch 등)를 사용합니다.
API 서버 (Gateway & File Service): S3의 Presigned URL 생성 기능과 Object Storage 기능을 모두 모방하는 중앙 서버입니다.
로컬 파일 시스템 (Storage): 실제 파일이 저장되는 서버의 하드 디스크 내 특정 폴더입니다.

2. 핵심 구현 매커니즘: Mock Presigned URL
AWS는 자체 비밀키로 URL을 서명합니다. 우리는 자체 JWT(JSON Web Token) 또는 HMAC(Hash-based Message Authentication Code)를 사용하여 이를 모방합니다.

서버가 생성할 Mock Presigned URL 예시:
https://localhost:5173/files/download/mock-file-key?token=eyJhbGciOiJIUzI1NiJ9...&expires=1678888888

mock-file-key: 파일이 저장된 로컬 경로를 유추할 수 있는 고유 키.
token: 서버만 아는 비밀키로 서명된 JWT. 이 안에 file_id, user_id, allowed_action(READ/WRITE), expires_at 정보를 담습니다.
expires: URL의 만료 시간 (보안성 확보).

3. 파일 업로드 흐름 (S3 Mocking)
사용자가 파일을 업로드하는 전체 프로세스입니다.

단계 1: Mock Presigned URL 요청 (Client -> Server)
사용자가 클라이언트에서 파일을 선택합니다. (예: image.png)
클라이언트는 서버의 API(POST /api/files/get-upload-url)에 파일 메타데이터(파일명, 타입)를 보냅니다.

단계 2: Mock Presigned URL 생성 및 레코드 생성 (Server)
서버는 PostgreSQL의 attachments 테이블에 상태를 'PENDING'으로 하여 파일 레코드를 생성하고 unique ID(file_uuid)를 발급합니다.
서버는 file_uuid, action='WRITE', expires=15m 정보를 담은 임시 토큰을 비밀키로 생성합니다.
서버는 이 토큰을 포함한 Mock Presigned URL을 구성하여 클라이언트에 반환합니다.

단계 3: 로컬로 파일 업로드 (Client -> Server)
클라이언트는 전달받은 Mock Presigned URL로 실제 파일 데이터를 PUT 또는 POST 요청으로 보냅니다. 이 요청은 여전히 S3가 아닌 우리 서버로 옵니다.

단계 4: 토큰 검증 및 로컬 저장 (Server)
서버의 파일 업로드 핸들러는 URL의 토큰을 먼저 해독하고 검증합니다.
서명이 유효한가?
만료되지 않았는가?
이 토큰이 'WRITE' 권한을 가지고 있는가?
검증이 성공하면, 서버는 요청 스트림에서 파일 데이터를 읽어 로컬 디렉토리 (예: /var/www/easydoc/uploads/file_uuid.png)에 직접 저장합니다.
저장이 완료되면 PostgreSQL의 해당 파일 레코드 상태를 'COMPLETED'로 업데이트합니다.

4. 파일 다운로드 흐름 (S3 Mocking)
파일을 안전하게 다운로드하거나 웹 페이지에 표시할 때의 프로세스입니다.

단계 1: Mock Presigned URL 요청 (Client -> Server)
클라이언트가 특정 파일의 다운로드 URL을 요청합니다. (GET /api/files/{file_uuid}/get-download-url)

단계 2: Mock Presigned URL 생성 (Server)
서버는 현재 사용자의 권한을 체크한 뒤, 다운로드용 Mock Presigned URL을 생성합니다.
토큰 내용: file_uuid, action='READ', expires=30m

단계 3: 파일 요청 (Client -> Server)
클라이언트는 전달받은 URL을 <a> 태그의 href로 사용하거나 <img> 태그의 src로 사용하여 파일을 요청합니다.

단계 4: 토큰 검증 및 파일 스트리밍 (Server)
서버는 토큰을 검증합니다 (action='READ').
검증 성공 시, 로컬 파일 시스템에서 해당 파일을 찾아 읽습니다.

서버는 HTTP 응답 헤더(Content-Type, Content-Disposition)를 설정하고, 파일 데이터를 클라이언트로 스트리밍(streaming)하여 보냅니다.

# 12 채널에 메시지 등록

현재 채널의 메시지를 계시글로 등록하는 것은 
채널 내용의 계시판 밑에 글을 넣고 확인 버튼 (비행기 모양)을 누르면된다.

from langchain_text_splitters import RecursiveCharacterTextSplitter

# 1. 원본 텍스트 준비
raw_text = {게시글 내용}

# 2. 텍스트 분할기 설정
text_splitter = RecursiveCharacterTextSplitter(
    chunk_size=config.rag.chunk_size,
    chunk_overlap=config.rag.chunk_overlap,
    separators=["\n\n", "\n", ".", " "] # 단락 -> 줄바꿈 -> 마침표 순으로 분할 시도
)
chunks = text_splitter.split_text(raw_text)

from sentence_transformers import SentenceTransformer

# 모델 로드 (Mac GPU/MPS 가속 활용)
embed_model = SentenceTransformer('BAAI/bge-m3', device='mps')

# 텍스트를 벡터로 변환
vectors = embed_model.encode(chunks)

import lancedb

db = lancedb.connect("./lancedb_data")
table = db.open_table("my_rag_table") # 기존에 만든 테이블 열기

# 저장할 데이터 형식 구성
data_to_save = []
for i, (text, vector) in enumerate(zip(chunks, vectors)):
    data_to_save.append({
        "vector": vector,
        "text": text,
        "metadata": {"chunk_id": i, "type": "manual_text"}
    })

# DB에 추가 (이 순간 '학습' 완료)
table.add(data_to_save)

context_text = "\n\n".join(context_list)

# 3. Gemma 4에게 답변 요청 (프롬프트 구성)

"""

response = client.chat.completions.create(model="gemma4:e4b", messages=[{"role": "user", "content": prompt}])

# 4. 답변 뒤에 Reference 붙여서 출력
print("--- Gemma의 답변 ---")
print(response.choices[0].message.content)

# 18.5.4 추가된 이벤트의 표현
- 이벤트가 추가되면 해당 날자에 표시가 된다.
- 일단위 보기
  . 하루 종일 옵션이면 하루 종일 이벤트 칸에 보인다.
  . 하루 종일 옵션이 아니면, 해당 시간대에 지정된 색상을 배경색으로하여서 박스로 나타나도록 한다. 
    단 배경색은 50% 투명으로 한다.
  . 여러개의 이벤트가 있어서 시간대가 겹치면, 이벤트의 크기를 가로를 1/n으로하여서 나타낸다.
    예를 들어서 2개의 이벤트가 겹치면 하나씩 반으로 나누어서 표시되도록 한다.
    예를 들어서 3개의 이벤트가 겹치면 1/3씩나누어서 표시 되도록 한다.

- 주단위 보기
  . 하루 종일 옵션이면 하루 종일 이벤트 칸에 보인다.
  . 하루 종일 옵션이 아니면, 해당 시간대에 지정된 색상을 배경색으로하여서 박스로 나타나도록 한다. 
    단 배경색은 50% 투명으로 한다.
- 월단위 표기
  . 그냥 이벤트 이름을 지정된 색상을 배경색으로 보여준다.

- 연단위 표기
  . 날자 배경색을 이벤트의 갯수에 맞추어서 투명도를 정한다.
  . 1개면 80% 투명
  . 2개면 60% 투명
  . 3개면 40% 투명
  . 4개면 20% 투명
  . 5개이상이면 0% 투명

# 21.3.1 Direcrt Message Display 영역의 표시 방식

   전송 된 메시지는 만화책의 사각형 말풍선 처럼 나타난다.
  - 내가 보낸 메시지는 Direct Message창의 오른쪽에서 타나나고
  - 상대방이 보낸 메시지는 Direct Message창의 왼쪽에서 타나난다.
 
# 21.3.2 말풍선 표시 방식

# 21.3.2.1 말풍선 표시 방식
  - 말풍선은 다음과 같은 요소가 표현된다. 
  1) 메시지를 보낸 사람의 이름 
  2) 메시지를 보낸 시간
  3) 메시지 내용 : TEXT
  4) 메시지 첨부 파일 : 최대 10개까지 첨부 된다. 
     - 첨부 파일은 모두 CLIP ICON과 파일 이름으로 나타난다. 
  5) 메시지 보낸 사람의 경우 말풍선 표시내용에 "수정"과 "삭제" 버튼이 있다.
     - 수정 버튼을 누르면 다시 메시지를 수정할 수 있다. 
     - 삭제 버튼을 누르면 메시지를 삭제 할 수 있다. 

# 21.3.2.2 상대방이 메시지를 읽으면 읽음 표시를 한다. 

  6) 상대방이 메시지를 읽으면 읽음 표시를 한다. 
     - 상대방이 1명이면 그냥 읽음 표시만 하면 된다.
     - 상대방이 여러명이면, 읽은 상대방 명수를 함께 표시한다.
     - 예를 들어서 1명이 읽으면 "읽음 1"로 표시한다.
     - 예를 들어서 3명이 읽으면 "읽음 3" 으로 표시한다. 
     - 읽음 상태를 표시하는 것은 반드시 보낸 사람 창에서만 표시된다.
     - 다른 사람들은 볼 수 없다. 
     - 읽지 않은 경우 "읽지 않음" 으로 표시해줘

# 21.3.2.2.1 읽음 상태 여부 위치
     
     - 읽음 상태 여부는 Direct 메시지 옆에 나타나도록 해줘

# 21.3.2.2.2 읽음 상태 표시 계정

     - 일금 상태 표시 계정은 메시지를 발신한 사람에게만 보인다.
     - 다른 사람에게는 보이지 않는다.

# 21.4.2.1 삭제된 메시지의 표현

  그리고 해당 메시지의 표시는 "메시지가 삭제 되었습니다." 로 표시된다.
  그리고 더이상 삭제 및 수정 옵션은 표시되지 않는다.

# DS.005 ATTACHMENTS (첨부파일)

파일의 실제 데이터(Binary)는 클라우드 스토리지(AWS S3, Google Cloud Storage 등)에 저장하고, DB에는 그 **경로(URL)**를 저장하는 것이 정석입니다.

- File ID     : 파일 고유 식별자,PK
- Post ID     : 연결된 게시글/댓글 ID,FK (어느 글의 파일인지 식별)
- Uploader ID : 업로드한 사용자 ID,FK 
- File Name   : 사용자가 올린 원본 파일명,예: report_final.pdf
- Stored Path : 실제 저장된 경로 URL,예: ~/ObjectFile/20260413/uuid_name.pdf
- File Type   : 파일 확장자/MIME 타입,"예: image/png, application/pdf"
- File Size   : 파일 크기,Byte 단위 (용량 제한 체크용)
- Created At  : 업로드 일시


파일이 첨부되면 파일마다, 이와 같은 정보를 만들어서 카산드라 DB에 저장한다.
원본 파일은 기존의 방식처럼 저장하고 저장된 위치에 대한 정보를 받아서 Stored Path 에 저장한다.
File Name 은 첨부된 파일의 원본 이름을 추가한다.

나중에 다운로드 할 때. 원본 이름으로 다운로드 하도록 한다.

첨부파일 하나마다 이 정보를 만들면, File ID를 해당 POST의 Attachments에 등록한다.


POST는 그 내용이 카산드라 DB에 등록된다.
ATTACMENTS도 마찬가지로 카산드라 DB에 등록된다.
다만, ATTACHMENTS에서 파일은 ObjectFile 폴더에 저장한다.
저장할 때 폴더의 이름은 File ID 로 생성한다.

ATTACMENTS 정보만 카산드라에 등록되며, 
POST에는 10개의 Attachments 슬롯이 있는데 이곳에 등록이 된다.
만약 10개보다 많은 파일이 첨부되면, 경고 메시지를 내고 진행하지 않는다.
10개의 파일은 각각 정리한다.

# DS.006 CALENDAR_EVENTS (캘린더 이벤트)

캘린더에 등록되는 일정/이벤트를 저장한다.
반복 일정, 하루 종일 일정, 참석자 초대, 채널 연동을 지원한다.

- Event ID        : 이벤트 고유 식별자, PK (Primary Key), UUID
- Title           : 이벤트 제목, 텍스트 (최대 200자)
- Description     : 이벤트 상세 내용, 텍스트 (선택)
- Location        : 장소 또는 회의 링크, 텍스트 (선택)
- Start At        : 이벤트 시작 일시, DateTime (타임존 포함)
- End At          : 이벤트 종료 일시, DateTime (타임존 포함)
- Is All Day      : 하루 종일 여부, Boolean (true이면 Start/End의 시간 무시)
- Recurrence      : 반복 규칙, 텍스트 (RRULE 형식, 예: FREQ=WEEKLY;BYDAY=MO,WE,FR / 없으면 NULL)
- Recurrence End  : 반복 종료 일시, DateTime (반복 없으면 NULL)
- Author ID       : 이벤트 생성자 ID, FK → USERS
- Attendee IDs    : 참석자 User ID 목록, 배열 (선택, 없으면 빈 배열)
- Channel ID      : 연동된 채널 ID, FK → CHANNELS (채널 공지 연동 시 사용, 선택)
- Color           : 이벤트 표시 색상, 텍스트 (hex 코드, 예: #4f46e5 / NULL이면 기본 색)
- Security Level  : 보안 등급 (0: 누구나, 1: 팀원, 2: 팀장, 3: 임원, 4: 대표이사)
- Is Edited       : 수정 여부, Boolean
- Created At      : 이벤트 생성 일시
- Updated At      : 최종 수정 일시
